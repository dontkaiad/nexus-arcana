## BUG #1 — Повторяющаяся задача не переходит в «В процессе» после первого выполнения

**Симптом.** Задача с повтором (например интервал «Каждые N дней») создаётся
со статусом `Not started`. Когда Кай нажимает «Сделано» в напоминании —
статус остаётся `Not started`, не переходит в `In progress`.

**Ожидаемое поведение.** Повторяющаяся задача никогда полностью не «Done» —
паттерн должен быть:
1. Создание → `Not started`.
2. Первое срабатывание + «Сделано» через бот → **`In progress`** (отметка что
   текущая итерация выполнена; задача продолжает повторяться).
3. Каждое следующее срабатывание остаётся в `In progress`.

**Связанный sub-bug (Mini App).** В «Мой день» → «Расписание» задача
продолжает висеть после нажатия «Сделано» через бот. Должна была уйти из
расписания на сегодня и появиться на следующую дату повтора.

**Где смотреть:**
- `nexus/handlers/tasks.py` — обработчик inline-кнопки «Сделано» на
  напоминании повторяющейся задачи. Пишет ли он `Статус = In progress`
  при наличии поля `Повтор`?
- `core/reminder_scheduler.py` — что происходит со статусом при tick'е
  повтора (создаётся ли новая итерация / обновляется существующая).
- `miniapp/backend/routes/today.py` — фильтр расписания на день. Проверяет
  ли статус, дату последнего выполнения, дату следующего повтора?
- Возможно нужен новый field в Notion: «Последнее выполнение» (Date) —
  чтобы Mini App мог скрыть задачу из расписания на сегодня после
  отметки даже если статус не Done.

**Объём.** Разведка ~30 мин (понять текущую логику callback'а «Сделано»),
фикс ~1-2 ч (статус-переход + Mini App фильтр + регрессы).

**Не делать ночью** — затрагивает три модуля и UX, нужна разведка с
адекватным состоянием Кай.

## BUG #2 — Память не использует existing связи (алиасы)

**Симптом.** В БД 🧠 Память существует запись с явным алиасом — формат:
`<сущность>_<тема>` + связь `<главное_имя>` + текст «у X краткая кличка
Y» (или аналогичная фраза-алиас).

Когда Кай позже пишет сообщение используя **алиас** (Y вместо X), бот
создаёт **новую запись с алиасом как главным именем**:
- ключ `<алиас>_<тема>`, связь `<алиас>` (вместо канонического `<главное_имя>`).

В результате память расщепляется: записи про одну сущность лежат под
двумя именами и не находятся при поиске по основному.

**Ожидаемое поведение.** Перед созданием новой записи `memory_save`:
1. Query existing `🧠 Память` по подстроке (тема + любая часть имени).
2. Если найдена запись формата «у X краткая кличка Y» (или аналог) —
   распознать алиас.
3. Канонизировать имя в новой записи: связь = `X`, ключ = `X_<тема>`.

**Где смотреть:**
- `core/memory.py` — функции `auto_suggest_memory`, `memory_save`. Сейчас
  при memory_save Haiku/Sonnet берёт имя из текста как есть, без проверки
  existing на алиасы.
- Промпт классификатора `memory_save` в `core/classifier.py` — не передаёт
  ли он existing memories в контекст? Скорее всего нет.
- Нужен **alias-resolver** на уровне `core/memory.py`: один call в Haiku
  с контекстом «вот существующие записи про эту тему/имя — есть ли
  алиасы которые надо канонизировать?» Возвращает либо канонический name
  либо None (создавать как написано).

**Архитектурно.** Это частный случай большей проблемы — Память не
использует свои же existing записи как контекст при создании новых.
Альтернатива — pre-search query по теме перед save.

**Объём.** Разведка ~1 ч (текущий путь memory_save, какие промпты задействованы),
фикс ~2-3 ч (alias-resolver + интеграция в save-pipeline + регрессы +
guard от ложных «алиасов»).

**Не делать ночью** — нужны living examples из Notion для тестирования
и осторожный rollout (легко сломать существующие связи).

## SANITY-DIAGNOSTIC #1 — отложено для следующей сессии

В рамках задачи #2 хорошо бы получить «срез» того что Haiku/Sonnet
видит и решает при memory_save с алиасом. Полезно прогнать:

```python
# core/memory.py — функция auto_suggest_memory или save-pipeline
# на тексте «<алиас> <тема>» (Кай знает реальный пример из её данных)
# — что Haiku возвращает? видит ли existing запись с алиасом?
```

Этот срез нужен ДО фикса — чтобы понимать, доходит ли до Haiku вообще
контекст existing memories, или промпт пустой/обрезанный. Без этого
фикс может промахнуться (закодить alias-resolver не там где нужно).

Не делать в авто-режиме — потребует API call с реальным именем из
жизни Кай (privacy guard) и должна делаться при ней.
